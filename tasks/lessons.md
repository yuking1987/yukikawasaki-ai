## 2026-07-16 案件コンテキスト蒸留（並列サブエージェント運用）
- **問題**: 複数ファイル担当のサブエージェントに任せたところ、1担当が「さらに孫担当を作り、孫は蒸留“本文テキスト”を返すだけでファイルに書かない」状態になり、context.md が作られなかった。
  - **なぜ**: 指示に「ファイルを書く」主体が曖昧だと、担当が“調整役”に回って作業を再委譲してしまう。孫は「本文を返す＝完了」と解釈する。
  - **どう直すか（次回ルール）**:
    1. 書き込みを伴う担当への指示は必ず「**あなた自身が Write/Edit でファイルを書く。サブエージェントを新たに作らない**」と明記する。
    2. 各担当の完了後は、**メイン側でファイルの実在をfilesystemで検証**する（`ls`/`grep -l`）。返事の文面だけで完了扱いしない。
- **書式**: サブエージェントの返答本文はしばしば `>` が `&gt;` にHTMLエスケープされる。ファイル化する際は `>` に直す（grep で `&gt;` 混入を最終チェック）。
- **機微情報**: 素材メールに本番FTP等の認証情報が平文で混じることがある。context.md には**値を転記せず「認証情報あり・取扱注意」の事実のみ**記す。最後に `ftp://user:pass@` 形式の漏洩を grep で確認する。
- **クライアント判定**: メール素材にはノイズが多い（相互リンク＝SEOバックリンク営業[backlink@]、ツール自動通知、営業メルマガ[配信停止リンク]、決済/SaaSサポート、協力/制作会社）。件数が多くても「制作/保守の実体があるか」で判定し、非該当は skip。
- **同一クライアントの別ソース**: フリーメール(gmail/ocn等)経由の素材が既存クライアントと同一のことがある（例: herb.ocn.ne.jp＝梅樹園、luck.ocn＝佐賀造船鉄工所）。**別ドメインでも署名の会社名で同一社か判定し、その会社の既存フォルダに集約**する（別フォルダを乱立させない）。ただし herb.ocn(梅樹園) と luck.ocn(佐賀造船) は**互いに別会社**なので混ぜない。

## 2026-07-16 オフィス表示：Codex専務の役割定義ミス（川崎さんフィードバック）
- **問題**: OfficeView の「Codex専務」バッジを、`status==="revision"`（要再考）の件数で「レビュー中 N」と表示していた。専務は**要再考の担当ではない**。
  - **正しい役割**: Codex専務＝**メンター役**。各社員が出した成果物のうち**重要なもの(importance:high)**を、Codexにレビュー・助言してもらう存在。要再考(revision)とは無関係。
  - **どう直したか**: バッジの母数を `importance==="high" && status==="pending" && review_status!=="passed"` に変更（[web/App.tsx] OfficeView の `codexReview`）。tooltipも「各社員の重要案件をレビュー・助言するメンター役」に修正。
  - **次回ルール**: キャラ/役割に紐づくUI指標を作るときは、**そのキャラの役割定義を先に確認**してから母数フィールドを選ぶ。`reviewed_by`/`review_status`/`importance` と `status(revision等)` は意味が別物。混同しない。
  - **未整備メモ**: `reviewed_by=codex` を自動で立てる処理は現状なし（GUI/API経由で記録する設計）。将来、高リスク案件を自動でCodexレビューに回すなら別途パイプライン実装が要る。

## 2026-07-16 追記：機微情報の扱い＆確認姿勢（川崎さんフィードバック）
- **機微情報は context.md に一切残さない（全般）**。パスワード値・Basic認証・スプレッドシート/DriveのURLとID・GitHubコミットハッシュ・請求書番号(INV…)・GA/UAプロパティID・**IPアドレス**・「パスワードの作り方（例：当日の年月日8桁）」まで、**ランダムな英数字の羅列と機微な仕組みはすべて削除**し「（値は別管理）」等に置換する。**既存の実例本文（メール全文）も対象**。
  - **なぜ**: 学習・流出リスク。平文の認証情報は絶対に含めない、が川崎さんの明確な要求。
  - **手順**: 蒸留/編集後、必ず grep で IP・google URL・INV・commit hash・パスワード値をスイープして残存ゼロを確認する。
- **判断に迷うことはすべて聞く（全般）**。特に「既存の上書き/統合可否」「クライアントか否かの境界」「機微情報の削除範囲」は独断で進めず AskUserQuestion で確認する。
- **報告文の書き方**: 複数の別会社を1行に並べて書くと「同一視した」と誤解される（例: 「herb=梅樹園、luck=佐賀造船 を統合」→川崎さんが“2社を混ぜた”と誤解）。**別会社は行を分けて明示**する。
- **照合インデックス**: `vault/70_references/clients-index.md`（ドメイン↔フォルダ↔正式名称↔宛名注意）を維持し、打ち返し前に照合する。名称ズレ（kkb=九州急行バス, sanrizkj=三立鑑定 等）はここで防ぐ。
