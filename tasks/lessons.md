## 2026-07-21 送信元(reply_to)の埋め直しで、取得範囲を広げて古いカードを大量に湧かせた（自戒）
- **背景**: 「今後カード化しない」ボタンは `reply_to` に送信元アドレスがある時だけ出る。機能追加(7/17 18:30)前に作られた居残りカードは `reply_to` が空でボタンが出なかった。取り込み(`ingest-imap.ts`)に、空カードを送信元で埋め直す処理を追加（新規カードは作らず既存だけ埋める、正しい設計）。
- **やらかし①**: 埋め直しを増やそうと `IMAP_SINCE_DAYS=120` で再取り込みした結果、**過去3〜6月の解決済みスレッドが新規カード12枚として「対応待ち」に湧いた**。ingestは「窓内の未カード化スレッドを新規化」もするため、窓を広げると埋め直しの副産物として大量の新規カードが生まれる。埋め直し目的で窓を広げてはいけない。
- **やらかし②**: 埋め直しの初版で、スレッド内に相手メッセージが無い時 `|| last` で最新（＝自分の送信）を送信元に採用し、`reply_to` に自分(kawasaki@gb-jp.com)を6枚に書いてしまった。`reply_to` は常に相手であるべき。相手が窓内に無ければ**埋めない**が正解（`inMsg = reverse().find(from!==ME)` で、見つからなければ空のまま）。
- **どう収束したか**: 誤って自分になった6枚は gray-matter で reply_to を除去。湧いた12枚は削除せず `updateStatus(..,'done')` で「対応済み」に。元メールがサーバから消えていたムームーの1枚だけは、同シリーズの別カード(7/20)にあった確定送信元 `news@service.muumuu-domain.com` を手で補填。
- **次回ルール**: (1) backfill/埋め直し目的でingestを走らせるときは既定窓(30日)のまま。窓拡大は「新規カードが増える」と理解した上でのみ。(2) 送信元の推定は「相手メッセージが取れた時だけ」。取れなければ空を許容し、自分アドレスや最新メッセージで代用しない。(3) vault変更は削除でなく状態遷移(done)で片付ける。[[maintenance-judgment]] 系と同じく、frontmatter直編集は gray-matter 経由で。

## 2026-07-21 Asana説明欄が「----」で切られ依頼本文が丸ごと消えていた（川崎さん指摘）
- **問題**: トコトン通知カード #105910（湯らっくす バナー削除＋Twitter表記変更）の「## 判定サマリ」が
  「件名のみ・実依頼は管理画面でしか見られない」となっていた。だがAsanaの説明欄には依頼内容
  （①福岡店バナー削除 ②Twitterをインスタの左にアイコン表記）が**はっきり書かれていた**。
- **原因**: `server/ingest-asana.ts` の `clean()` の `SIG_CUT` に `-{4,}` / `={4,}`（4個以上の区切り線）が
  入っており、**最初の区切り線で本文をカット**していた。トコトン等の定型通知は「【概要】＋区切り線の"内側"に
  ◇項目・①②…の依頼本文」が入る形式なので、最初の `----` で切ると概要が丸ごと消える。これは
  [[lessons.md のメール `----`＝署名で本体を捨てた事故]] と**同じ種類**（区切り線＝署名という思い込み）。
- **被害範囲**: トコトン由来カード全部（梅北/蔵本/九州商船/真宗大谷派/こどもらぼ/ケイティウィン/湯らっくす）で
  概要が空。コメントがある案件は会話で中身が補われて被害が見えにくく、コメントの無い新規依頼（#105910）で露呈した。
- **どう直したか**: 区切り線単体をカット条件から外し、署名/フッタは**文字マーカー**で切る方針に統一。
  CIY署名は元々「人材の定着」「プライバシーマーク」の文字で判定しているので区切り線を外しても漏れない。
  トコトンのフッタは行頭記号ゆれ（**※/＊/*** など＝実カードは全角 `※` だった）を吸収する `[＊*※•・]?\s*このメールは`
  と `.*トコトンシステムから自動送信`（行の途中に出ても拾う）で確実に落とす。末尾の装飾線・空行は `DIVIDER_ONLY` で除去。
  実データ形状のテストで①②が残り・CIY署名が消えることを確認（`scratchpad/test-clean.mjs`）。
- **過去分の復元**: `server/backfill-asana-notes.ts`（新規・GETのみ／`--write`で反映）で既存カードの説明欄だけを
  再取得・再cleanで差し替え（frontmatter/コメント/添付/判定サマリ/ドラフトは触らない・冪等）。11件を復元（トコトン全件＋
  区切り線で切れていたJML/GB/ciy-biz等）。footer残0・YAML健全を検証。#105910 は判定サマリも実依頼で作り直した。
- **部品の再利用**: `clean` を export し backfill から使えるようにしたが、`ingest-asana.ts` は末尾で無条件に `main()` を
  呼んでいたため import で本番取り込みが走る危険があった。`fs.realpathSync(argv[1])===realpath(import.meta.url)` の
  **直接実行ガード**を追加（macOSの `/var`↔`/private/var` シンボリックリンクで単純比較は一致しないため realpath 必須）。
- **表示側の改行**: スレッド先頭（`thread-lead`＝件名＋説明欄）を素の `<div>{lead}</div>` で描画しており改行が潰れて
  一続きに見えていた。本文と同じ `<SimpleMarkdown>` に揃えて改行を保持（[web/App.tsx] ThreadView）。
- **次回ルール**: 本文クリーナーで「区切り線＝ここから下は不要」と決めつけない。定型通知は**区切り線の内側が本体**。
  署名/フッタは"線"でなく**固有の文字マーカー**で切り、行頭記号のゆれ（※/＊/*）まで吸収する。取り込み系を触ったら、
  コメントの無い新規1件で概要が残るかを必ず確認する（コメント有りは被害が隠れる）。他ソースの `cleanBody` も同観点で点検。
  モジュール末尾の `main()` は**直接実行ガード**で囲み、import 副作用で本番処理が走らないようにする。
  表示テキストは改行を保持する描画（SimpleMarkdown/pre-wrap）を通す。生の `{text}` を div に入れない。

## 2026-07-21 AIの草案 vs 本人の実返信の「食い違い」を学習に取り込む（川崎さんフィードバック）
- **指摘**: 川崎さんはAIの草案をそのまま使わず、別の解釈・言い回しで打ち返すことが多い。
  「少し表現を変えただけ」のこともあるが、そこはAIの判断と本人の解釈の不一致が最も濃く出る場所。
  それを更新時に吸収して学びにできないか、という要望。
- **見落としていた設計**: 取り込み(`ingest-imap.ts`)は「実際に送った返信」を正例(`_memory/replies.md`)に
  貯めていたが、**AIの草案は比較されず捨てていた**。正解だけ集めて、"食い違いそのもの"は毎回消していた。
  `extractDraft()` はコメントに「差分記録用」とありながら返信の場面で未使用だった＝部品はあり繋いでいないだけ。
- **どう直したか**: 対応済みで閉じる直前に、既存カードの草案と実返信を突き合わせ、
  `_memory/draft-vs-sent.md` に記録（`appendDraftVsSent`）。`textSimilarity`（文字bigram Dice）で
  0.7以上=「微修正」/未満=「要学習」とタグ付けし、"無視して書き直した"ケースだけ浮かせる。
  草案づくりのプロンプト(`gb-draft.sh` 手順2.5)で、この「要学習」を毎回参照して同じズレを避ける。
- **次回ルール**: 学習パイプラインを作るときは「正解の蓄積」だけで満足しない。
  **AIの出力と本人の実採用の"差分"**が最良の教師データ。承認/送信の瞬間に草案を捨てず、
  実物と突き合わせて残す。ルールへの昇華は無人化せず声かけ時（既存方針を踏襲）。

## 2026-07-21 相互リンクのお礼は「やります」でなく「やりました＋実物URL」で即完了（草案 vs 実返信・昇華第1号）
- **教師データ**: `_memory/draft-vs-sent.md` が自動で捕まえた最初の「要学習」（ヴィスタス様／相互リンク・類似度66%）を、
  声かけ時ルールに昇華したもの。手順2.5は「同じ客」しか照合しないため、相互リンク全般に効く一般ルールとして記録する。
- **食い違いの中身**（お礼2段落までは草案＝実返信で一致。分かれたのはその後）:
  - 私の草案: 「これから弊社ブログでご紹介させていただきます。例：/medias/6362（過去の別記事）。内容・掲載時期は
    社内調整のうえ改めてご連絡、今しばらく…。アンカーテキストは弊社の通例に合わせご了承ください」
    ＝**未来形の約束＋例URLでお茶濁し＋先延ばし＋相手に条件/宿題を残す**。
  - 川崎さんの実返信: 「弊社でも貴社にご紹介いただいた旨、**記事作成させていただきました**。/medias/6883（実際に作った新記事）。
    ご確認ください」＝**もう作って公開済み＋実物URL＋簡潔に完了**。
- **なぜズレたか**: AIは記事という成果物を実際には作れないので、「例URL＋未来形＋保留」に逃げた。結果、川崎さんの実運用
  （相互リンクのような軽い定型は"その場で片付けて完了報告"）と真逆の「先延ばし・宿題残し」の印象になった。
- **次回ルール**:
  1. 相互リンク・お礼・軽微な定型対応は「対応します／社内調整のうえ後日ご連絡」と**先延ばししない**。川崎さんの標準は
     **やりました型（完了報告）**。ドラフト本文は未来形の約束でなく「〜させていただきました。URL：（作成後に差し込み）」の完了型にする。
  2. AIは成果物を作れないので、**「例：過去URL」で未来形の約束を濁さない**。代わりに「## 判定サマリ」に
     「相互リンク記事を1本作成・公開し、そのURLをドラフトに差し込む（川崎さん作業）」と**やるべき実作業を明示**して渡す。
  3. **余計な条件・お願いを足さない**（アンカーテキストの通例・掲載時期の調整など）。相手に保留や宿題を残さず、簡潔に完了で閉じる。
- **横展開**: 未対応の相互リンクカード（Gigantic/松揚/Tabicolle/センタリング/Starlight 等）は、次の「✨今すぐ生成」で
  この完了型に置き換わる想定。[[tone_external]] にも通じる（過剰な謝り・条件付けをしない、簡潔・前向き）。

## 2026-07-17 support@ 宛の取り込み漏れ／転送メール／Excel添付

- **取り込み対象は「どの箱に届くか」で決まる。宛先だけ見て納得しない**。support@gb-jp.com 宛のメールが

  - **次回ルール**: 保険コードは**実データで効くことを確認できないなら入れない**。
    動いているように見えて効いていない状態を残さない。

- **転送メール(Fwd)は、依頼の中身が丸ごと捨てられていた**（今回の本丸）。`cleanBody` は

  - **どう直したか**: `forwardedBody()` で「---------- 転送されたメール ----------」以降を
    引用記号を外して本文として復元（`threadBody` に組込み）。
  - **次回ルール**: 本文整形の規則を書くときは、**Fwd と Re を分けて考える**。Re の引用は冗長だが、

  - **次回ルール**: 添付の中身を扱うときは「文字に起こす」で満足しない。**指示は絵で来る**前提で、

## 2026-07-16 vault frontmatter の手動編集でYAMLを壊し重複カードを生成（自戒）
- **やらかし**: item .md の `thread_last_id` を python の `re.sub(r'^thread_last_id:.*$', ...)` で1行だけ置換した。だが値が複数行の折り畳みスカラー（`thread_last_id: >-` ＋次行にインデントされた値）だったため、2行目が孤児になりYAMLが不正化。`listItems`(gray-matter)がそのカードを読めず、ingestが既存カードに照合できずに**重複カードを新規生成**した。
